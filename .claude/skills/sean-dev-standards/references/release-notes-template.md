# Release Notes 模板

以下是标准 release notes 模板。生成时使用实际数据替换所有占位符。

---

## 文件命名

```
YYYY-MM-DD-feature-name.md
```

**示例：** `2026-07-07-page-view-analytics.md`

---

## 完整模板

```markdown
# Release Notes: [功能名称]

**日期：** YYYY-MM-DD
**分支：** `branch-name`
**状态：** ✅ 已合并至 master / 🚧 开发中 / 📦 已发布

---

## 思考过程

> 记录架构设计中的关键思考和决策，每条说明「做了什么」+「为什么这样做」+「trade-off」。

- **决策 1：** 做了什么 — 为什么这样做 — trade-off
- **决策 2：** 做了什么 — 为什么这样做 — trade-off
- **决策 3：** 做了什么 — 为什么这样做 — trade-off

### 关键 trade-off

| 决策 | 分类 | 理由 |
|------|------|------|
| 决策描述 | 性能/复杂度/可维护性/... | 选择理由和放弃的选项 |
```

## 填写说明

### 需求说明
- 1-2 句话讲清楚背景和目的，不要写成长篇大论
- 核心功能用编号列表，每个功能点一句话
- 方案选择如果有「造 vs 买」的决策，务必记录

### 思考过程
- 重点记录**为什么这样做**，而不是仅仅罗列做了什么
- Trade-off 表格是核心——让后来者理解你放弃了什么、为什么
- 分类标签建议：性能、复杂度、可维护性、成本、时间

### 实现过程
- 任务分解如实列出，标记每个任务的状态
- Critical 问题必须记录，说明发现者和修复方案

### 指标
- **代码量**：用 git diff --stat 获取实际数据
- **文件清单**：按目录分组，每个文件一句话说明用途
- **测试覆盖**：如实填写，没有测试就写 0，不编造
- **Token 使用量**：适用于 AI 辅助开发，从工具通知中获取
- **用时**：区分实际耗时和计划预估，便于后续校准
- **提交树**：`git log --oneline` 获取，按时间倒序

### 决策审计追踪
- 如果已有独立计划文档，引用即可，不用重复粘贴
- 没有计划文档时，至少记录 3-5 个关键决策

### 已知限制
- 诚实列出当前版本不做的事情，避免后来者踩坑
- 每个限制附上原因（时间不够、技术不成熟、范围外等）

### 后续事项
- 用 checkbox 格式，方便后续迭代直接当作 checklist 使用
- 优先级高的放前面
