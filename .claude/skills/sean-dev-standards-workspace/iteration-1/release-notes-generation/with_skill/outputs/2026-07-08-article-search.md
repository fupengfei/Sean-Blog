## 需求说明

> 为博客前台提供文章搜索能力，用户可通过关键词快速检索文章，支持标题和正文内容匹配。

### 核心功能

1. **关键词搜索** — 支持对文章标题和正文内容进行关键词匹配
2. **搜索 UI** — 前台提供搜索栏和搜索结果展示页面
3. **空结果处理** — 无匹配结果时展示友好提示

### 决策：搜索引擎选型

**选择 MySQL LIKE 查询。** 理由：当前数据量小（< 1000 篇文章），引入 Elasticsearch 会显著增加运维复杂度和资源开销。待文章量增长到万级别后再评估是否迁移至 Elasticsearch。

## 思考过程

> 记录架构设计中的关键思考和决策，每条说明「做了什么」+「为什么这样做」+「trade-off」。

- **决策 1：使用 MySQL LIKE 而非 Elasticsearch** — 数据量不足 1000 篇时，MySQL LIKE 配合合理索引即可满足性能需求，引入 ES 意味着额外的部署、监控和数据同步成本 — trade-off：牺牲了全文检索的分词能力和相关性排序，换取更简单的运维和更低的资源消耗

- **决策 2：搜索结果默认按更新时间降序排列** — 用户搜索时更关心最新内容，按更新时间降序符合博客场景的阅读习惯 — trade-off：未实现按相关度排序（MySQL LIKE 不提供相关度评分），可能在某些场景下最新结果并非最相关

### 关键 trade-off

| 决策 | 分类 | 理由 |
|------|------|------|
| MySQL LIKE 替代 Elasticsearch | 复杂度 / 成本 | 数据量 < 1000，ES 的运维和资源成本不值得；后续可迁移 |
| 按更新时间降序而非相关度排序 | 用户体验 | MySQL LIKE 不提供相关度评分，更新时间降序是合理折中 |

## 已知限制

- **不支持分词和模糊匹配** — MySQL LIKE 仅做简单的子串匹配，用户必须输入连续的关键词。原因：未引入全文检索引擎
- **未实现搜索高亮** — 搜索结果中不标记匹配位置。原因：v1 范围外，后续迭代考虑
- **无搜索历史 / 热门搜索** — 当前不记录用户搜索行为。原因：需要额外的数据表和分析逻辑，v1 优先保证核心搜索可用
- **性能上限** — 当文章量超过万级别时 LIKE 查询可能变慢。原因：未使用倒排索引，后续需迁移至 Elasticsearch
